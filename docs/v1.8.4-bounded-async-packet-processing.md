# EdgeNetSwitch v1.8.4 — Bounded Asynchronous Packet Processing and Pressure Visibility

## Thesis
> A queue does not make the system faster. It makes the system honest.

This release introduces bounded asynchronous packet handling to expose pressure explicitly instead of masking it through synchronous callback coupling.

## Problem Statement
The previous `PacketProcessor` executed processing directly in the subscriber callback path. Ingress and processing were therefore serialized by construction: packets could not meaningfully accumulate, admission never saturated, and overload could not be represented as an explicit runtime state.

Under stress, throughput looked stable only because ingress was implicitly paced by processing latency. This hid backpressure, removed backlog visibility, and prevented causal overload attribution.

## Design Change
`PacketProcessor` now owns:
- internal `std::deque<Packet>` work queue
- mutex protecting queue access
- worker thread running `processLoop()`
- bounded capacity `MAX_QUEUE_SIZE = 1024`

`PacketRx` no longer performs packet processing. It now performs admission:
- If queue size is below bound, enqueue and return.
- If queue is full, emit `PacketDropped(reason = QueueOverflow)`.

Processing executes asynchronously in `processLoop()`. A simulated processing delay remains to model CPU/parsing cost under sustained load.

Semantically, this introduces explicit operating regions that were previously collapsed into one callback path:
- burst absorption while queue headroom exists
- saturation when capacity is reached
- recovery while backlog drains

## Why `QueueOverflow` Matters
Overload loss is a distinct failure mode and must not be conflated with parse errors, policy drops, or malformed packet handling. A single undifferentiated drop counter destroys root-cause resolution.

`QueueOverflow` as a first-class `PacketDropped` reason provides:
- causal attribution of loss to admission saturation
- actionable telemetry for overload policy tuning
- a stable contract for future backpressure and rate-control behavior

Without this reason code, operators see loss but cannot determine whether to optimize parsing cost, policy, queue bounds, or ingress control.

## Observability Fields
`PacketStats` now includes:
- `ingress_packets`
- `processed_packets`
- `processing_gap`
- `drops_by_reason`

These counters separate ingress, service, and loss:
- `ingress_packets` tracks offered load.
- `processed_packets` tracks completed work.
- `processing_gap` captures unresolved work over the observation window.
- `drops_by_reason` attributes loss causally.

`processing_gap` is meaningful only with reasoned drops. A rising gap with low overflow often indicates transient backlog growth; rising `QueueOverflow` indicates sustained admission failure. Throughput alone cannot distinguish those conditions.

`ControlDispatch` now exposes these fields in both text and JSON packet-stats output, including structured drop-reason distribution. Text supports immediate operator inspection; JSON supports dashboards, alerting, and automated overload diagnosis.

## Testing Implications
Asynchronous semantics invalidate synchronous-era immediate assertions. State convergence is no longer instant after a `PacketRx` event; tests must validate eventual outcomes within bounded deadlines.

Updated tests use wait loops with timeout-based termination, then assert expected counter states. This avoids scheduler-timing flakiness and validates queue/overflow behavior when offered load exceeds service rate relative to `MAX_QUEUE_SIZE`.

## Limitations and Future Refinement
The current worker loop is intentionally minimal and should not be represented as performance-final. It establishes correctness and visibility first.

Current limitations:
- wake behavior is basic and polling-oriented
- idle efficiency and wake latency are not optimized
- simulated delay is a workload model, not an optimized pipeline

Planned refinement is straightforward: replace polling-style wakeups with `condition_variable`-based signaling on enqueue while preserving bounded admission, explicit `QueueOverflow`, and current telemetry contracts.

## Capacity Boundary

`MAX_QUEUE_SIZE` defines the system’s admission boundary.

This is not an internal implementation detail. It is the explicit limit of how much work the system is willing to accept before enforcing loss.

Once the queue reaches this bound:
- new packets are not delayed
- new packets are not buffered elsewhere
- new packets are dropped immediately with `QueueOverflow`

This guarantees that latency does not grow unbounded and that overload is expressed as loss rather than hidden delay.

## Conclusion
v1.8.4 corrects a core architectural blind spot. By decoupling ingress from processing with a bounded queue, EdgeNetSwitch now surfaces backlog, saturation, and overload loss as explicit states rather than implicit callback slowdown.

This makes pressure measurable and actionable. It unlocks concrete next work: real backpressure policy, overload control, queue-aware scheduling, and packet switching logic that can reason about admitted versus dropped traffic under load.

## Runtime Validation and Pressure Observation
Use three terminals: one for daemon runtime, one for UDP load generation, one for control-plane queries.

### 1. Start the daemon
```bash
cd /Users/togunchan/Desktop/Projelerim/EdgeNetSwitch
./build/EdgeNetSwitchDaemon
```

### 2. Generate UDP load from a separate terminal
Steady traffic (moderate, controlled ingress):
```bash
for i in $(seq 1 5000); do
  printf 'id=%s;payload=steady\n' "$i" | nc -u -w0 127.0.0.1 9000
  sleep 0.002
done
```

Burst traffic (admission pressure test):
```bash
seq 1 20000 | xargs -I{} -P16 sh -c \
  'printf "id={};payload=burst\n" | nc -u -w0 127.0.0.1 9000'
```

The burst case drives arrival rate above worker service rate (including simulated processing delay), forcing queue growth and then overflow once `MAX_QUEUE_SIZE` is reached.

### 3. Query packet stats through the control plane
CLI wrapper (recommended):
```bash
./build/EdgeNetSwitchDaemon packet-stats:json
```

Direct protocol over UNIX socket:
```bash
echo "1.2|packet-stats:json" | nc -U /tmp/edgenetswitch.sock
```

Poll repeatedly during load:
```bash
while true; do
  ./build/EdgeNetSwitchDaemon packet-stats:json
  sleep 0.5
done
```

### 4. What to observe
- `ingress_packets`: total packets admitted into `PacketRx` path; should rise with offered traffic.
- `processed_packets`: packets drained by `processLoop()` and emitted as processed; bounded by worker throughput.
- `processing_gap`: `ingress_packets - processed_packets`; emerges when ingress outpaces processing and collapses only if processing catches up. `processing_gap` is not a defect; it is the observable backlog of admitted but unprocessed work.
- `drops["queue_overflow"]`: - `drops["queue_overflow"]`: increases when new packets arrive while the queue is already at capacity at admission time; this represents explicit overload loss, not parser/validation loss. `QueueOverflow` is not a failure; it is the system enforcing its capacity boundary.

Expected behavior by regime:
- No load: `ingress_packets` and `processed_packets` stable, `processing_gap = 0`, `drops["queue_overflow"] = 0`.
- Moderate load: both counters increase; small transient `processing_gap` may appear and drain; overflow drops remain zero.
- Overload: `ingress_packets` continues rising, `processed_packets` increases at worker-limited slope, `processing_gap` grows, then `drops["queue_overflow"]` starts increasing as queue admission saturates.

This behavior was impossible before v1.8.4 because synchronous in-callback processing had no independent backlog state and no bounded admission boundary. The new model exposes pressure as observable state transitions instead of hiding it as callback path latency.
