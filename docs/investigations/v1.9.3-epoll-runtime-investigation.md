# EdgeNetSwitch v1.9.3 — Epoll Runtime Investigation

## 1. Investigation Scope

This investigation reruns the UDP ingress comparison after the v1.9.3 epoll
integration. The earlier v1.9.2 results were collected with verbose packet-path
logging enabled, and later testing showed that logging was large enough to move
burst throughput from the low thousands of packets per second into the hundreds
of thousands. In this test family, logging configuration affected measured
throughput more than the ingress model.

The older debug-mode numbers are still useful historical context, but they are
not a fair baseline for judging the runtime model. This report measures the
three ingress models again under the same `warn` logging configuration:

- blocking `recvfrom()`
- non-blocking polling `recvfrom()`
- epoll readiness dispatch

The goal of this investigation is to evaluate whether the v1.9.3 epoll 
integration can replace busy polling while maintaining acceptable latency, 
correctness, and runtime stability.

The short answer from this run is:

- logging dominated the earlier throughput comparison
- with equivalent `warn` logging, all three models landed in the same burst
  throughput class
- blocking `recvfrom()` was slightly faster in raw burst throughput in this
  single-UDP-socket benchmark
- non-blocking polling still had the lowest periodic latency, but it produced
  extreme idle polling
- epoll removed busy polling, preserved lifecycle correctness, improved
  shutdown behavior, and gives EdgeNetSwitch the right foundation for a
  multi-file-descriptor Linux runtime

Measured runtime data, code inspection, and future profiling hypotheses are kept
separate. This report does not claim hard realtime behavior or kernel causality
beyond what was measured.

## 3. Methodology

Each run used a fresh daemon process, a fresh control socket, and the same
`warn` logging configuration. Runtime samples were collected through
`packet-stats:json` over the UNIX control socket while traffic was active.

Periodic traffic used the requested command shape:

```bash
while true; do
    printf "id=1;payload=test\n"
    sleep 0.1
done | nc -u 127.0.0.1 9000
```

The same command was run with `sleep 0.05` and `sleep 0.01`.

Burst traffic used:

```bash
while true; do
    printf "id=1;payload=test\n"
done | nc -u 127.0.0.1 9000
```

Periodic tests collected six snapshots at 2 second intervals. Burst tests
collected six snapshots at 5 second intervals over approximately 30 seconds.

The tables use three snapshot terms consistently:

- `mean` is the average across all collected snapshots
- `final` is the value observed in the final snapshot of the run
- `max` is the highest value observed during the run

The tables report `average_processing_latency_ns`, `max_processing_latency_ns`,
`processing_gap`, `duplicate_events`, polling idle count or epoll UDP
drain-completion count, packet-rate metrics, and burst-only queue/pending
metrics where applicable.

`rx_packets_per_sec` is EWMA-smoothed packet rate. `rx_packets_per_sec_raw` is the unsmoothed packet rate calculated over the current measurement window.

Metric terminology:

- `processing_gap`: difference between ingress events and completed terminal outcomes
- `pending_terminal_events`: lifecycle records observed at ingress that have not
  yet reached a terminal outcome; terminal outcomes are `PacketProcessed` and
  `PacketDropped`
- `idle_polls`: polling iterations that reached `EAGAIN`/`EWOULDBLOCK` without
  receiving a packet
- `udp_drain_completions`: successful epoll receive-queue drains that ended at
  `EAGAIN`/`EWOULDBLOCK`; not busy polling

readiness. Non-blocking polling reacts quickly when packets arrive, but it pays
for that by repeatedly calling `recvfrom()` while no packet is ready. Epoll does

busy idle loop.

Blocking `recvfrom()` produced the highest average raw burst rate in this
single-socket run. Epoll had the smallest maximum `processing_gap` and
`pending_terminal_events`. All three models reported `queue_overflow` drops
under sustained overload, and duplicate terminal events did not appear.

The burst result is not that epoll is faster. It shows epoll remained
competitive with the `recvfrom()`-based models while avoiding busy polling and
keeping the runtime structure needed for coordinated multi-FD behavior.

## 6. Correctness Validation

The correctness checks were stable across the valid measurement set. No
duplicate terminal events appeared, and periodic processing did not fall behind
publication. Under burst load, processing gaps remained bounded, with the epoll
run showing the smallest maximum gap in this set.

No periodic run reported drops. Burst runs reported explicit `queue_overflow`
drops for all three runtime models. That is the expected accounting path for
overload: drops are visible and attributed, not hidden inside a packet-rate
counter.

All measured runs continued to respond to `packet-stats:json`. The historical
blocking and non-blocking variants required forced termination in this harness
because their older control-thread architecture did not shut down
deterministically. The epoll runtime exited cleanly on `SIGINT` in every
measured run. That result follows from the v1.9.3 design: blocking waits are
owned by explicit readiness and wakeup paths.

## 7. Analysis

### Throughput And v1.9.2 Comparison

The controlled rerun changes the throughput interpretation. Historical v1.9.2
documentation reported much lower debug-mode burst throughput, strong periodic
latency for non-blocking polling, and tens of millions of `idle_polls`. Those
numbers were collected under a different logging configuration, so direct
throughput comparison against this `warn`-mode rerun would be misleading.

With `warn` logging and Release builds, the three ingress models are much closer
under burst load. Blocking `recvfrom()` was slightly ahead in average raw burst
rate, and non-blocking polling and epoll were close enough that this run is best
interpreted as same-class throughput rather than a decisive ranking.

The remaining throughput ceiling is probably in packet-path work after
readiness delivery, not in epoll itself. That remains a profiling target, not a
conclusion from this benchmark alone.

### Latency

Periodic latency still favors non-blocking polling. The tables show polling
consistently below the blocking and epoll models, while blocking and epoll sit
in the same broad range. Burst average latency converged for all models. That
is expected from the runtime's point of view: under sustained traffic, there is
less idle time between packet arrivals and processing work stays hot.

The tradeoff is clear. Non-blocking polling buys the lowest periodic latency by
spending CPU aggressively when idle. Epoll gives up that busy-poll latency
advantage, but it removes the idle spin and keeps latency in a low
hundreds-of-microseconds range for these periodic tests.

The remaining epoll periodic latency gap may involve scheduler wakeup cost and
the existing packet handoff path. No kernel tracing was collected here, so that
is a future profiling hypothesis.

### Epoll Behavior

This part is based on code inspection of the current epoll implementation, tied
back to the measurements where possible.

#### UDP Queue Draining

`UdpReadyHandler::onEvent()` dispatches readiness to
`UdpReceiver::processReadableEvent()`. The current `processReadableEvent()`
loops up to `MaxPacketsPerWakeup = 256` and exits when `handleReadable()`
returns `NoData`. `NoData` is returned when non-blocking `recvfrom()` reaches
`EAGAIN` or `EWOULDBLOCK`.

The current epoll UDP path drains the receive queue after `EPOLLIN`, bounded by
a 256 packet per-wakeup budget. This is a material difference from the earlier
single-read readiness path.

#### Remaining Hot-Path Candidates

Code inspection identified several remaining packet-path contributors:

- logging message construction
- `MessagingBus` publication
- `PacketStats` accounting
- packet allocation and copying
- `PacketProcessor` queue handoff

These components were not isolated during this investigation and remain future
profiling targets. The measurements show correct accounting and bounded backlog
under the tested workloads; they do not attribute standalone cost to any one
component.

## 8. Architectural Decision

The burst table should not be read as a raw-throughput contest in isolation. A
single UDP socket naturally favors a dedicated blocking receiver, and blocking
`recvfrom()` slightly outperformed epoll in average burst throughput here.

The broader result is that epoll stayed in the same throughput class, preserved
lifecycle correctness, avoided idle polling, and exited cleanly through the
shutdown wakeup path. Those properties matter more for EdgeNetSwitch than a
small single-socket burst advantage, because the runtime is moving toward
coordinated Linux event sources behind one deterministic dispatch mechanism.

## Appendix A — Metric Reference

| Metric | Meaning |
| --- | --- |
| `processing_gap` | Difference between ingress events and completed terminal outcomes |
| `pending_terminal_events` | Lifecycle records awaiting `PacketProcessed` or `PacketDropped` |
| `idle_polls` | Empty polling iterations that reached `EAGAIN`/`EWOULDBLOCK` |
| `udp_drain_completions` | Successful epoll receive-queue drains ending at `EAGAIN`/`EWOULDBLOCK` |
| `queue_overflow` | Packets dropped because processing capacity was exceeded |
| `duplicate_terminal_events` | Duplicate `PacketProcessed`/`PacketDropped` observations for the same lifecycle |
