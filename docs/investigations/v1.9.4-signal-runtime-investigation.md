# EdgeNetSwitch v1.9.4 Signal-Aware Runtime Shutdown Investigation

## Objective

v1.9.4 investigated how EdgeNetSwitch should record process termination
signals while preserving the existing deterministic shutdown sequence.

The immediate case was local `Ctrl+C`, which sends `SIGINT` on Linux. The
service case was `SIGTERM`, expected when EdgeNetSwitch later runs as a
`systemd` service in a QEMU / Yocto environment.

Scope:

- distinguish developer interrupt shutdown from service-stop shutdown
- expose the shutdown reason in runtime logs
- keep signal handling inside a narrow signal-safe boundary
- validate both `SIGINT` and `SIGTERM`
- avoid changing the runtime architecture

## Initial Problem

The runtime could already stop through `Ctrl+C`, but the shutdown path did not
record why shutdown was requested. A developer interrupt, a service stop, and
future shutdown initiators were indistinguishable in the shutdown state and log
path.

That was an observability gap for daemon operation. The process should still
exit through the existing controlled shutdown sequence, but the reason should
be visible when reading logs or correlating runtime behavior with host-level
actions.

## Shutdown Reason Modeling

The first step was to model shutdown causes explicitly in `ShutdownReason.hpp`
with a `ShutdownReason` enum class.

Implemented reasons:

- `Unknown`
- `ControlPlaneRequest`
- `SignalInterrupt`
- `SignalTerminate`

Signal mappings:

- `SIGINT` -> `SignalInterrupt`
- `SIGTERM` -> `SignalTerminate`

This replaced an unstructured stop condition with a typed shutdown origin.

## Shutdown Request Design

The next step was a `ShutdownRequest` object with three responsibilities:

- request shutdown with a reason
- report whether shutdown was requested
- report the stored shutdown reason

The first design used two atomics:

- `std::atomic<bool> requested_`
- `std::atomic<ShutdownReason> reason_`

The reason was data-race avoidance. In C++, a data race is undefined behavior,
and shutdown state may be observed by runtime code while signal-related state
changes occur.

Mutex-based synchronization was considered inappropriate for the signal path.
`mutex`, `lock_guard`, and `unique_lock` should not be used inside a signal
handler.

## Concurrency Considerations

The relevant execution contexts were:

- main thread
- epoll thread
- asynchronous signal handler execution

The main thread owns the daemon loop and checks for shutdown. The epoll thread
owns readiness processing and is stopped through the existing shutdown path.

The first integration replaced the previous `g_stopRequested` style state in
`main.cpp` with `ShutdownRequest`. After that change, `Ctrl+C` still produced
a clean shutdown. This validated the request object as a fit for the daemon
loop, but it did not settle whether the signal handler itself should mutate
runtime shutdown state.

## Runtime Integration

Shutdown orchestration stayed in `src/daemon/main.cpp`. The daemon loop checks
the shutdown request while continuing telemetry ticks, health checks, status
publication, and sleep cadence.

This work did not replace the epoll shutdown architecture. It only added a
typed reason to the existing shutdown decision and log path.

## Shutdown Reason Visibility

`ShutdownReason` was made loggable through a `toString` function returning
`std::string_view`. The function returns views over fixed string literals
instead of allocating new `std::string` objects for reason names.

`SIGINT` validation produced:

```text
[2026-06-20 20:45:56][WARN] Stop requested. Shutting down...
[2026-06-20 20:45:56][INFO] [SHUTDOWN] Reason: SignalInterrupt
[2026-06-20 20:45:56][INFO] RuntimeStatus: state=STOPPING uptime_ms=1569 tick_count=15
```

This confirmed that shutdown was not only clean, but also reason-visible.

## Signal-Safe Boundary Investigation

The first signal-aware design used a process-level pointer:

```cpp
ShutdownRequest *g_shutdownRequest = nullptr;
```

That made it possible for the signal handler to update the runtime shutdown
object directly. The design was later corrected: the handler should not perform
runtime orchestration or touch complex runtime objects.

The corrected boundary uses:

```cpp
volatile sig_atomic_t g_receivedSignal = 0;
```

The signal handler records only the received signal number. Normal runtime code
then observes that value and updates `ShutdownRequest`.

The final signal path is:

```text
SIGINT / SIGTERM
  -> sigaction-installed handler
  -> volatile sig_atomic_t g_receivedSignal
  -> main runtime loop
  -> ShutdownRequest::request(...)
  -> ShutdownReason::SignalInterrupt or ShutdownReason::SignalTerminate
  -> shutdown reason logged
  -> existing deterministic shutdown sequence stops the epoll event loop
```

This is the main architectural correction from the investigation.

## Volatile and sig_atomic_t Findings

Production daemon examples commonly use a signal-visible variable shaped like:

```cpp
volatile sig_atomic_t g_signal = 0;
```

The finding was specific:

- `sig_atomic_t` is the intended type for simple signal-handler-visible values.
- `volatile` prevents the compiler from treating the value as stable across
  reads in the runtime loop.
- `volatile` does not make a variable atomic.
- `volatile` is not a general thread-synchronization primitive.

In the final implementation, the handler writes only the signal number. The
richer shutdown state remains in normal runtime context.

## Shutdown Latency Investigation

Shutdown responsiveness was tested by changing `tick_ms` from `100` to `5000`
and then pressing `Ctrl+C`.

Observed sequence:

```text
[2026-06-21 14:32:27][INFO] telemetry_export: uptime_ms=2 tick_count=1

Ctrl+C

[2026-06-21 14:32:32][INFO] [SHUTDOWN] Stopping epoll event loop
```

Finding:

```text
shutdown latency can be approximately equal to tick_ms
```

The signal handler records the signal immediately, but the daemon acts on it
only when the main loop reaches the next tick iteration. v1.9.4 did not add an
immediate wakeup for the main loop sleep.

## signal() vs sigaction()

The signal registration model was changed from `signal()` / `std::signal` to
`sigaction`.

The reason was Linux daemon control. `sigaction` explicitly controls:

- handler function selection
- signal masking while the handler runs
- interrupted system-call restart behavior
- optional extended signal information

v1.9.4 does not use all of those capabilities, but `sigaction` is the clearer
Linux daemon model and leaves the installed `SIGINT` / `SIGTERM` handlers
explicit.

## SIGINT Validation

Validation method:

```text
Ctrl+C
```

Result:

```text
SignalInterrupt
```

Representative log:

```text
[2026-06-20 20:45:56][WARN] Stop requested. Shutting down...
[2026-06-20 20:45:56][INFO] [SHUTDOWN] Reason: SignalInterrupt
[2026-06-20 20:45:56][INFO] RuntimeStatus: state=STOPPING uptime_ms=1569 tick_count=15
```

The runtime exited cleanly through the normal daemon shutdown path.

## SIGTERM Validation

`SIGTERM` was validated because it represents the expected service-stop path.

Command:

```bash
kill -TERM <pid>
```

Result:

```text
[SHUTDOWN] Reason: SignalTerminate
```

This confirmed that service-style termination is distinguishable from local
developer interruption.

## Final Architecture

The final architecture is intentionally small:

- signal handlers are installed through `sigaction`
- the handler only records `SIGINT` or `SIGTERM` into `g_receivedSignal`
- the main daemon loop converts that value into a `ShutdownRequest`
- `ShutdownRequest` stores the typed reason
- `ShutdownReason::toString` exposes the reason in logs
- epoll shutdown still follows the existing deterministic shutdown path

No new runtime owner, reactor redesign, or alternative shutdown architecture
was introduced.

## Findings

- `SIGINT` and `SIGTERM` need distinct runtime reasons.
- A stop boolean was insufficient for shutdown observability.
- `ShutdownReason` provides a typed shutdown-origin vocabulary.
- `ShutdownRequest` provides the request/query surface for shutdown state.
- The first direct signal-handler-to-runtime-object design was corrected.
- The final signal handler records only the signal number.
- Mutex-based synchronization is not appropriate inside the signal handler.
- `volatile sig_atomic_t` is suitable for this minimal handoff.
- `volatile` does not provide atomicity or general thread safety.
- `std::string_view` is sufficient for fixed shutdown reason names.
- Shutdown latency is currently bounded by the daemon loop cadence.
- With `tick_ms=5000`, shutdown proceeded when the next tick observed the
  recorded signal.
- `sigaction` is the preferred Linux daemon registration model over
  `signal()` for this runtime.
- `SIGINT` validation produced `SignalInterrupt`.
- `SIGTERM` validation produced `SignalTerminate`.

## Conclusions

v1.9.4 made EdgeNetSwitch shutdown reason-aware without replacing the runtime
shutdown architecture.

The investigation started with typed shutdown reasons and an atomic
`ShutdownRequest`, validated clean `Ctrl+C` shutdown, then corrected the signal
boundary so the handler only records the signal and normal runtime code updates
shutdown state.

Both required signal paths were validated:

- `SIGINT` -> `SignalInterrupt`
- `SIGTERM` -> `SignalTerminate`

The main limitation is shutdown latency. The signal is recorded immediately,
but the runtime acts on it only when the main loop wakes for the next tick. In
the `tick_ms=5000` experiment, shutdown proceeded roughly five seconds after
the last tick log.

The final result is a practical daemon improvement: signal causes are
preserved, deterministic shutdown remains in place, service-stop behavior is
validated, and the signal handler stays inside a narrow signal-safe boundary.
