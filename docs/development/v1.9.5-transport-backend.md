# EdgeNetSwitch v1.9.5 Development Notes

# Initial Observations

At this stage, `SwitchForwardingEngine` was only responsible for producing a `ForwardingDecision`.

```cpp
ForwardingDecision processPacket(
    const Packet& packet,
    std::uint32_t ingress_port,
    std::uint64_t tick);
```

However, this decision was not yet translated into an actual packet transmission.

After reviewing the codebase, the current execution flow looked like this:

```

UDP Packet
↓
PacketProcessor::processPacket()
↓
SwitchForwardingEngine::processPacket()
↓
ForwardingDecision
↓
ForwardingDecisionMade (MessageBus)
↓
Logger

```

For example, when the forwarding engine decided that a packet should be sent to Port 3, the runtime simply logged that decision. No real transmission was performed.

The `ForwardingDecisionMade` message was published to the `MessagingBus`, and the only subscriber produced the following log:

```cpp
Logger::info("ForwardingDecisionMade: lifecycle_id=" +
             std::to_string(event->lifecycle_id) +
             " action=" + action +
             " egress_ports=[" + ports + "]");
```

After reviewing the runtime flow, I concluded that packet dispatch should happen inside `PacketProcessor::processPacket()` **before** publishing the `ForwardingDecisionMade` message.

The new execution order would become:

```

ForwardingDecision
↓
Transport / Port Dispatch
↓
ForwardingDecisionMade (MessageBus)
↓
Logger

```

This way, the forwarding decision would be executed first, and only afterwards would the runtime publish observable events such as logs or telemetry.

---

# Introducing TransmitResult

To represent transport execution results consistently, I introduced a new model:

```

TransmitResult.hpp

```

The file contains an enum class describing possible transport outcomes:

```cpp
enum class TransmitStatus
```

and a result structure containing:

- status
- port_id
- bytes_transmitted
- native_error

Every transport backend can now return the same unified result model.

---

# TransportManager

The next step was introducing a runtime component responsible for executing forwarding decisions.

For this purpose I implemented `TransportManager`.

Two public functions were added:

- `registerBackend()`
- `transmit()`

`registerBackend()` stores transport implementations using:

```cpp
std::unordered_map<
    std::uint32_t,
    std::unique_ptr<PortBackend>>
```

The key represents the runtime port identifier.

Whenever

```cpp
transmit(port_id, packet)
```

is called, the corresponding backend is located and its own `transmit()` implementation is executed.

---

# First Backend: VirtualPortBackend

`PortBackend` itself is an abstract interface.

Before implementing real Linux networking, I wanted to verify the architecture using a minimal backend.

For this reason I created:

```

VirtualPortBackend

```

Its implementation simply returns

```cpp
TransmitStatus::Success
```

without performing any real network transmission.

To verify that the runtime actually reached this backend, I temporarily added the following log:

```cpp
Logger::info("VirtualPortBackend: transmit() called");
```

# First Validation Attempt

Initially I tried injecting a packet directly through UDP:

```bash
echo "id=1;payload=hello" | nc -u 127.0.0.1 9000
```

The expected transport log never appeared.

After investigating the forwarding engine I noticed the following validation:

```cpp
if (!packet.source_mac || !packet.destination_mac)
    return {};
```

The packet sent through `netcat` only contained an ID and a payload.

Since both MAC addresses were missing, the packet was rejected before the forwarding stage.

Instead of crafting UDP packets manually, I realized I already had a deterministic packet injection mechanism available through the control plane.

```bash
echo "1.2|send-packet:broadcast" | nc -U /tmp/edgenetswitch.sock
```

This command creates a synthetic packet containing valid source and destination MAC addresses.

---

# Missing TransportManager

Even after using the synthetic packet injection mechanism, the expected transport log still did not appear.

After reviewing the runtime initialization I found the problem.

`PacketProcessor` had been constructed with a null transport manager.

```cpp
PacketProcessor packetProcessor(
    bus,
    &forwardingEngine,
    nullptr,
    failureInjector);
```

As a result,

```
transport_manager_
```

was never available.

I fixed the runtime initialization as follows:

```cpp
transport::TransportManager transportManager;

transportManager.registerBackend(
    1,
    std::make_unique<transport::VirtualPortBackend>());

transportManager.registerBackend(
    3,
    std::make_unique<transport::VirtualPortBackend>());

transportManager.registerBackend(
    4,
    std::make_unique<transport::VirtualPortBackend>());

transportManager.registerBackend(
    5,
    std::make_unique<transport::VirtualPortBackend>());

PacketProcessor packetProcessor(
    bus,
    &forwardingEngine,
    &transportManager,
    failureInjector);
```

# Designing UdpPortBackend

With the architecture validated, the next step was implementing a real Linux UDP transport backend.

I introduced:

```

UdpPortBackend

```

derived from `PortBackend`.

The primary design goal was that every backend should own and manage its own transport resources.

The backend therefore stores:

```cpp
FileDescriptor socket_;
sockaddr_in destination_{};
```

Using `FileDescriptor` preserves the RAII model already used throughout the runtime.

It also allows every transport socket to participate in the existing file descriptor tracking mechanism.

The constructor therefore receives:

```cpp
UdpPortBackend(
    const UdpEndpoint& endpoint,
    FdRegistry* fd_registry);
```

The constructor creates the socket:

```cpp
::socket(AF_INET, SOCK_DGRAM, 0)
```

and immediately transfers ownership into the RAII wrapper:

```cpp
socket_(
    ::socket(AF_INET, SOCK_DGRAM, 0),
    fd_registry,
    FdType::UdpSocket);
```

Because of this, every newly created socket is automatically registered by `FdRegistry`.

The constructor verifies successful creation:

```cpp
if (!socket_.valid())
{
    throw std::system_error(...);
}
```

It then prepares the destination address:

```cpp
destination_.sin_family = AF_INET;
destination_.sin_port = htons(endpoint.port);
```

Finally, `inet_pton()` converts the textual IP address into the binary representation expected by the Linux networking stack.

```cpp
inet_pton(
    AF_INET,
    endpoint.ip.c_str(),
    &destination_.sin_addr);
```

If the address is invalid, construction fails immediately.

```cpp
throw std::invalid_argument(...);
```

I intentionally preferred this approach because a transport backend that cannot create its socket or prepare its destination should never exist in a partially initialized state.

---

# UdpPortBackend::transmit()

With construction complete, I implemented the actual transmission routine.

The function first checks whether the payload is empty.

```cpp
if (packet.payload.empty())
```

In that case, the function simply reports a successful transmission of zero bytes.

Otherwise, packet transmission is performed using Linux's `sendto()` system call.

```cpp
::sendto(
    socket_.get(),
    packet.payload.data(),
    packet.payload.size(),
    0,
    reinterpret_cast<const sockaddr*>(&destination_),
    sizeof(destination_));
```

If transmission fails, the native Linux error code (`errno`) is returned inside the transport result.

```cpp
return {
    .status = TransmitStatus::SendFailed,
    .native_error = errno
};
```

Otherwise the backend reports success together with the transmitted byte count.

```cpp
return {
    .status = TransmitStatus::Success,
    .port_id = port_id_,
    .bytes_transmitted = ...
};
```

At this point `VirtualPortBackend` remained purely an architectural validation tool, while `UdpPortBackend` became the first backend capable of sending packets through the Linux UDP stack.

---

# First Linux Validation

Once `UdpPortBackend` was complete, I wanted to verify that it could actually transmit packets through the Linux networking stack.

Inside `main.cpp`, I replaced one of the virtual backends with a UDP backend.

```cpp
transportManager.registerBackend(
    1,
    std::make_unique<transport::UdpPortBackend>(
        1,
        transport::UdpEndpoint{"127.0.0.1", 9101},
        &fd_registry));
```

This meant that whenever the forwarding engine selected Port 1, the runtime would now use a real UDP socket.

## Validation Setup

I opened a second terminal and started listening on UDP port 9101.

```bash
nc -u -l 9101
```

With the daemon running, I injected another synthetic broadcast packet.

```bash
echo "1.2|send-packet:broadcast" | nc -U /tmp/edgenetswitch.sock
```

Initially this confused me.

Seeing

```
control-plane-broadcast
```

appear in the listening terminal made me think that the control command itself had been sent over UDP.

That is **not** what actually happens.

The `-U` option tells `netcat` to communicate through a Unix Domain Socket.

The command only reaches the runtime's `ControlServer`.

Inside the daemon, `handleSendPacket()` invokes `publishSyntheticPacket()`, which creates a synthetic `Packet` containing valid source and destination MAC addresses.

The packet is then injected into the runtime through:

```cpp
bus.publish(MessageType::PacketRx);
```

From this point onward, `PacketProcessor` does not know whether the packet originated from a real UDP receiver or from the control plane.

Every `PacketRx` message follows exactly the same processing pipeline.

The forwarding engine generates a `Flood` decision.

`PacketProcessor` then executes:

```cpp
transport_manager_->transmit(port, processedPacket);
```

`TransportManager` locates the previously registered backend and invokes:

```
UdpPortBackend::transmit()
```

which finally executes Linux's `sendto()` system call.

Meanwhile,

```bash
nc -u -l 9101
```

is simply listening on UDP port 9101.

When the kernel delivers the datagram, `netcat` prints its payload.

```
control-plane-broadcast
```

This validation was an important milestone.

For the first time, a packet generated entirely through the Control Plane was processed by the Data Plane and transmitted through the real Linux UDP networking stack.

The complete execution flow became:

```
Control Command
        ↓
ControlServer
        ↓
publishSyntheticPacket()
        ↓
MessagingBus
        ↓
PacketProcessor
        ↓
SwitchForwardingEngine
        ↓
ForwardingDecision
        ↓
TransportManager
        ↓
UdpPortBackend
        ↓
Linux sendto()
        ↓
UDP Socket (9101)
```

At this point, the runtime was no longer producing forwarding decisions only—it was capable of executing those decisions through a real Linux transport backend.
