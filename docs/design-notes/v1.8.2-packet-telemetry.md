# EdgeNetSwitch v1.8.2 — Packet Rate Telemetry with EWMA Smoothing

## 1. Problem Definition
Naive per-snapshot rate calculation is operationally unstable under deterministic runtimes that execute on non-real-time schedulers. In a 100 ms tick loop, packet arrival and snapshot timing are not phase-locked; the measured delta window can capture either a sparse segment or a burst boundary. This produces high variance in instantaneous rates even when long-run throughput is unchanged.

Even in a deterministic tick-driven runtime, packet arrival is not synchronized with the tick boundary. This leads to phase misalignment between event arrival and measurement windows, so adjacent windows can observe materially different event counts without any underlying change in sustained traffic level.

In practice, three sources dominate:
- Tick-to-arrival misalignment: packet bursts can land near window edges and inflate/deflate one interval.
- Scheduler noise: thread wake-up jitter shifts effective sampling windows.
- Burst traffic shape: queue-less packet streams are naturally clumpy at sub-second timescales.

This is effectively a sampling aliasing problem, where the sampling window does not align with the underlying signal frequency, producing inconsistent measurements.

Without filtering, control-plane telemetry oscillates between local peaks and troughs and is not reliable for trend interpretation.

## 2. Design Approach
Packet rate telemetry in v1.8.2 uses a time-gated update model inside `PacketStats`:
- Rate state updates only when elapsed time since last rate update is `>= 1000 ms`.
- Instantaneous rates are computed from counter deltas over that elapsed window.
- Two metric families are maintained and exported:
  - Raw rates (`rx_packets_per_sec_raw`, `rx_bytes_per_sec_raw`)
  - Smoothed rates (`rx_packets_per_sec`, `rx_bytes_per_sec`)

The 1-second gate decouples rate updates from the 100 ms runtime tick cadence. This eliminates high-frequency recomputation noise while keeping update frequency operationally useful for control-plane inspection.

## 3. EWMA Behavior in Runtime Systems
EWMA is used as a behavioral filter, not as a ground-truth estimator. With `alpha = 0.2`, the signal prioritizes stability over immediate responsiveness.

Operational implications:
- Lower alpha increases temporal inertia and suppresses burst-induced variance.
- Higher alpha tracks transients faster but reintroduces volatility.
- EWMA output necessarily lags step changes in true traffic level.

EWMA values should be interpreted as trend telemetry for system state transitions, not as an instantaneous measurement replacement.

EWMA introduces a controlled loss of temporal precision in exchange for improved observability stability.

## 4. Observability Model
Exporting both raw and smoothed rates is required for production debugging:
- Raw rate exposes what the delta window observed at update time.
- Smoothed rate exposes a stable control signal for trend evaluation.

This dual-view model supports:
- Validation of smoothing behavior against real event dynamics.
- Detection of pathological jitter (raw unstable, smoothed stable).
- Detection of traffic regime changes (raw shift sustained until smoothed converges).

Single-channel reporting (only raw or only smoothed) is insufficient for root-cause isolation.

## 5. CLI Output Example
```text
OK
rx_packets=321
rx_bytes=2353
rx_packets_per_sec=12
rx_bytes_per_sec=88
rx_packets_per_sec_raw=9
rx_bytes_per_sec_raw=76
END
```

Interpretation:
- `rx_packets` and `rx_bytes` are cumulative counters.
- `*_raw` fields report the most recent gated delta-rate.
- EWMA fields (`rx_packets_per_sec`, `rx_bytes_per_sec`) show filtered trend level.
- If EWMA is above raw, the system is decaying from a recent higher-load period.
- If raw is above EWMA, the system is entering a higher-load period.
- The divergence between raw and EWMA represents the temporal gradient of the traffic signal.

## 6. Test Methodology (CRITICAL)
Use daemon-side UDP ingress and control-plane query in parallel. These tests intentionally introduce burst patterns and phase misalignment to expose instability in raw rate calculations.

Stable traffic test:
```bash
while true; do
  printf "id=1;payload=test\n"
  sleep 0.05
done | nc -u 127.0.0.1 9000
```

Expected behavior:
- `rx_packets` and `rx_bytes` increase monotonically.
- `rx_packets_per_sec_raw` oscillates within a narrow band due to window alignment noise.
- `rx_packets_per_sec` converges to a stable plateau and drifts slowly.

Burst traffic test:
```bash
while true; do
  for i in {1..20}; do
    printf "id=1;payload=test\n"
  done
  sleep 0.2

  for i in {1..2}; do
    printf "id=1;payload=test\n"
  done
  sleep 0.2
done | nc -u 127.0.0.1 9000
```

Expected behavior:
- Raw rates show distinct high/low interval swings.
- EWMA rates suppress interval spikes and produce a slower envelope.
- During repeated burst cycles, EWMA centers around long-run effective throughput instead of instantaneous extremes.

## 7. Key Observations
- EWMA reduces variance, not error.
- Smoothed metrics lag behind real traffic transitions by design.
- Raw metrics reflect delta-window truth but are sensitive to timing noise.
- Cumulative counters remain the canonical integrity signal; rates are derivative telemetry.

## 8. Design Trade-offs
- Responsiveness vs stability is explicitly controlled by alpha (`0.2` favors stability).
- A 1-second gate lowers computational churn and avoids tick-level volatility at the cost of update latency.
- No blocking, locking, or allocation is introduced in the rate path.
- Counter ingestion remains atomic and non-blocking; telemetry derivation remains deterministic relative to call sequence.

## 9. Conclusion
Packet rate telemetry in v1.8.2 is a telemetry interpretation layer, not a measurement layer.  
Raw metrics expose observed interval behavior; EWMA metrics provide control-plane stability for operational reasoning under scheduler jitter and bursty traffic.  
It explicitly trades temporal accuracy for operational readability.

## 10. When Not to Use EWMA
EWMA is not suitable for:
- Real-time control decisions
- Rate limiting or protection logic
- Latency-sensitive feedback loops

In such cases, raw metrics or alternative aggregation strategies should be preferred.
