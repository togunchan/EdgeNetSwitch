# EdgeNetSwitch v1.9.2 — Blocking vs Non-Blocking Runtime Analysis

## Investigation Scope

v1.9.2 begins the blocking versus non-blocking runtime analysis phase.

The investigation starts from the UDP ingress path using blocking `recvfrom()`
behavior, then compares it with non-blocking polling. It records the first
ingress-to-processed latency baseline for packets that reach `PacketProcessed`
and captures the behavior visible through EdgeNetSwitch runtime metrics under
moderate and sustained ingress pressure.

The analysis is limited to runtime-visible timing and lifecycle behavior. It
does not claim hard realtime guarantees or explain OS scheduling decisions.

The runtime components in scope are:

- UDP ingress socket
- `MessagingBus`
- `PacketProcessor`
- packet lifecycle accounting
- structured control plane
- packet lifecycle invariants
- runtime timing instrumentation
- runtime-visible packet metrics

## Baseline Objective

The objective is to measure processed-packet latency through the existing
blocking ingress runtime and then compare it with non-blocking polling behavior.

The baseline is meant to answer four runtime questions:

- What is the ingress-to-processed latency under moderate periodic traffic?
- What happens to that latency under sustained ingress pressure?
- Does lifecycle completion behavior remain stable while ingress pressure is
  high?
- What timing effects are visible through process-level metrics?

This document treats ingress-to-processed latency as an observable runtime
signal, not as an OS trace.

## Instrumentation Added

### TimeUtils

`TimeUtils` now exposes:

```cpp
std::uint64_t nowNs()
```

The implementation uses `std::chrono::steady_clock`.

The purpose is high-resolution monotonic runtime latency measurement for packets
that reach `PacketProcessed`. The clock source is used to compare runtime event
boundaries inside the process.

### Packet

`Packet` now carries:

```cpp
std::uint64_t ingress_timestamp_ns
```

The timestamp records when the runtime first observes a successfully received
packet after socket ingress. It is used only for packets that reach
`PacketProcessed`; dropped paths are not included in this latency measurement.

### UdpReceiver

Immediately after successful `recvfrom()`:

```cpp
const auto ingress_ts = nowNs();
```

The timestamp is attached to the runtime packet before publication into the
messaging path.

### PacketStats

`PacketStats` now records:

- `total_processing_latency_ns_`
- `max_processing_latency_ns_`
- `latency_samples_`

Latency is measured from successful UDP ingress observation to
`PacketProcessed` completion. `PacketDropped` paths do not participate in the
processing-latency sample.

Average and maximum latency are exported through:

- `packet-stats`
- `packet-stats:json`

This keeps latency reporting on the same control-plane path as lifecycle and
drop accounting.

`latency_samples_` counts packets that have contributed to the latency
aggregate. During concurrent observation it may temporarily differ from
`processed_packets` if a snapshot lands between processing completion and the
latency-accounting update. Small divergence is expected in this runtime model.

## Traffic Generation

### Periodic Ingress

```bash
while true; do
  printf "id=1;payload=test\n"
  sleep 0.05
done | nc -u 127.0.0.1 9000
```

Purpose: measure steady-state ingress-to-processed latency under moderate
traffic.

The periodic baseline also used the same command shape with `sleep 0.1` and
`sleep 0.01` to measure lower and higher periodic ingress rates.

### Sustained Burst Ingress

```bash
while true; do
  printf "id=1;payload=test\n"
done | nc -u 127.0.0.1 9000
```

Purpose: observe runtime behavior under sustained ingress pressure.

## Periodic Ingress

Periodic ingress was measured with three sleep intervals. The goal was to see
how the blocking runtime behaved as packets arrived more continuously, without
turning the test into a sustained burst.

- `sleep 0.1`: approximately 8-9 packets/sec, average latency approximately
  447-452 microseconds, maximum latency approximately 0.75 milliseconds
- `sleep 0.05`: approximately 14-15 packets/sec, average latency approximately
  421-428 microseconds, maximum latency approximately 0.88 milliseconds
- `sleep 0.01`: approximately 58-60 packets/sec, average latency approximately
  334-337 microseconds, maximum latency approximately 0.80 milliseconds
- `processing_gap` remained zero
- `duplicate_events` remained zero
- no drops were observed

The periodic runs exercise the blocking ingress path without saturating the
processing path. Timing metrics remained stable, runtime processing remained
deterministic, and latency spikes stayed below 1 millisecond in these runs.

As the sleep interval decreases, the runtime spends less time between packets.
The lower average latency at higher periodic rates is consistent with the
runtime spending less time idle between packets. The maximum observed latency is
what this environment showed during these runs; it is not a timing guarantee.

## Sustained Burst Ingress

Measured sustained burst metrics:

- average ingress-to-processed latency stabilized around 86-95 microseconds
- maximum ingress-to-processed latency ranged approximately 4.7-6.5 milliseconds
- raw throughput stabilized around 20k packets per second
- `duplicate_events` remained zero

A representative earlier burst snapshot used for gap interpretation was:

```json
{
  "average_processing_latency_ns": 95047,
  "max_processing_latency_ns": 10102875,
  "ingress_packets": 280966,
  "processed_packets": 280965,
  "processing_gap": 1,
  "duplicate_events": 0
}
```

The same run family also showed:

- average ingress-to-processed latency decreased significantly under sustained
  load
- later maximum latency spikes remained in the approximately 4.7-6.5
  millisecond range
- runtime remained operational under burst ingress
- no duplicate lifecycle completion events occurred
- lifecycle completion accounting remained stable
- parse errors remained explicit and observable

The burst run changes the runtime operating pattern. The processing path has
less idle time between packets, and the lower average latency is consistent with
the runtime staying active more continuously.

The improved average does not mean the runtime has eliminated scheduling
variability. Later sustained burst runs still showed millisecond-scale tail
latency. Those spikes remain process-level evidence of timing jitter in the
observed execution window.

## Processing Gap Interpretation

Transient non-zero `processing_gap` values are expected during concurrent
ingress and processing.

Observed transient values included:

- `1`
- `4`
- `12`
- `15`

A control-plane read is not a single global snapshot of the runtime. While
ingress and processing continue at the same time, the read can catch packets
after ingress accounting and before `PacketProcessed` completion. Temporary
divergence between `ingress_packets` and `processed_packets` does not imply
packet loss or lifecycle corruption.

In the observed runs, processing_gap returned toward zero as processing caught
up with ingress traffic. It also matters that `duplicate_events` remained zero
while processing counters continued to advance under sustained ingress pressure.

## Parse Error Accounting

Parse failures emit `PacketDropped` before `PacketRx` publication.

As a result, `terminal_events` may exceed `ingress_packets` by the number of
parse-error drops. This is expected behavior in the current accounting model and
should not be interpreted as duplicate lifecycle completion.

For this investigation, the relevant correctness signal is that duplicate
terminal events remained zero, parse failures remained explicitly observable,
and completion behavior remained deterministic under load.

## Runtime Interpretation

Average latency dropped as ingress became more continuous. That is consistent
with the runtime spending less time idle between packets.

Tail latency still matters because the slowest observed completions did not move
the same way as the average. Burst ingress lowered average latency further, but
also exposed larger maximum latency.

These metrics do not explain OS internals. They show the timing behavior the
runtime could observe from inside the process.

## Invariant Preservation Under Load

Under sustained ingress, the main correctness checks stayed stable:

- duplicate terminal events remained zero
- lifecycle completion accounting remained stable
- the runtime stayed operational while offered ingress remained high

Parse errors also remained explicit and visible through the existing accounting
path. Load changed latency behavior, but it did not corrupt lifecycle completion
state in these runs.

## Non-Blocking Polling Investigation

The runtime was then switched from:

```cpp
IngressMode::Blocking
```

to:

```cpp
IngressMode::NonBlocking
```

The UDP socket was configured with `O_NONBLOCK` through `fcntl()`.

In this mode, `recvfrom()` no longer waits for a packet. When no packet is
available, it returns `EAGAIN` or `EWOULDBLOCK`. A new `idle_polls` metric was
added to count loop iterations where `recvfrom()` returned no packet.

During idle periods with no ingress traffic, `idle_polls` increased very
quickly. Observed values included approximately 34 million, 40 million, and 55
million iterations over short windows while no packets were being processed.
That made the busy polling cost visible in normal runtime metrics.

### Periodic Ingress Under Non-Blocking Polling

Measured periodic ingress behavior under non-blocking polling:

| Traffic Regime | Observed PPS | Average Accepted Latency | Max Accepted Latency | Correctness State |
| --- | ---: | ---: | ---: | --- |
| Low periodic ingress (`sleep 0.1`) | approximately 8-9 | approximately 52-53 us | approximately 0.14 ms | no drops, `processing_gap = 0`, `duplicate_events = 0` |
| Moderate periodic ingress (`sleep 0.05`) | approximately 15-16 | approximately 46-47 us | approximately 0.13 ms | no drops, `processing_gap = 0`, `duplicate_events = 0` |
| Higher periodic ingress (`sleep 0.01`) | approximately 58-60 | approximately 39 us | approximately 0.57 ms | no drops, `processing_gap = 0`, `duplicate_events = 0` |

These runs showed noticeably lower average latency than blocking mode. The mode
comparison is simple:

| Runtime Mode | Average Latency | CPU Efficiency | Runtime Behavior |
| --- | --- | --- | --- |
| Blocking | higher | efficient | worker sleeps while waiting |
| Non-blocking polling | lower | poor | worker continuously polls `recvfrom()` |

The `idle_polls` metric shows the cost: while idle, polling keeps calling
`recvfrom()`. This gives a useful baseline before adding readiness-based models
such as `epoll`.

## Final Comparison Summary

The investigation exposed a clear runtime tradeoff between blocking ingress and
non-blocking polling.

Blocking ingress produced higher average latency during periodic traffic, but
the runtime stayed CPU-efficient because the worker thread slept while waiting
for packets.

Non-blocking polling reduced average ingress-to-processed latency noticeably.
The runtime reacted to packets more quickly because `recvfrom()` no longer
blocked waiting for ingress.

The lower latency came with a visible runtime cost. During idle periods, the
worker thread continuously called `recvfrom()`, causing `idle_polls` to grow
rapidly even while no packets were arriving.

| Runtime Mode | Latency Profile | CPU Efficiency | Wakeup Model | Operational Cost |
| --- | --- | --- | --- | --- |
| Blocking | higher average latency | efficient | kernel blocking wakeup | lower idle CPU usage |
| Non-blocking polling | lower average latency | poor | continuous polling loop | aggressive idle spinning |

The investigation establishes the architectural motivation for readiness-based
event handling.

Blocking ingress conserves CPU time but delays wakeups. Non-blocking polling
improves responsiveness but wastes CPU cycles while idle. The next phase will
investigate whether `epoll` can preserve low-latency wakeups without the
busy-polling behavior observed in this phase.

## Next Investigation

v1.9.2 made the basic tradeoff visible. Blocking ingress kept idle CPU behavior
efficient but had higher periodic latency. Non-blocking polling reduced latency
but spent CPU aggressively when no packets were available.

The next step is a readiness-based ingress model, starting with `epoll`. That
should let the runtime wait for socket readiness instead of continuously polling
`recvfrom()`, while still measuring ingress-to-processed latency, CPU behavior,
and control-plane visibility under periodic and sustained traffic.

The correctness requirements stay the same: explicit FD ownership, observable
descriptor state, deterministic lifecycle accounting, explicit parse-failure
visibility, and no duplicate terminal completion events.
